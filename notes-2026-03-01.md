# Sessions Tool — Feature Plan

## Overview

A new tool called **Sessions** to be placed in the Tool Shed under the **Musician Tools** section. The purpose is to record and archive live jam sessions and band rehearsals. Users can record live audio directly from their device microphone, or upload previously recorded audio files. All recordings are stored on Amazon S3, consistent with how the app handles other file uploads.

---

## Audio Format Recommendation

**Recommended format: M4A (AAC audio in an MPEG-4 container)**

Rationale:
- **iOS**: `AVAudioRecorder` natively outputs AAC/M4A — no conversion needed.
- **Android**: `MediaRecorder` natively supports AAC output in an MPEG-4 container — no conversion needed.
- **Web**: The browser `MediaRecorder` API outputs WebM/Opus on Chrome/Firefox and MP4/AAC on Safari. Because Safari is the primary web browser on iPhone, and because we support both formats at the upload endpoint, the backend should accept both and (optionally) normalize to M4A on the server side using ffmpeg.
- **Playback**: M4A/AAC plays natively in all modern browsers and on all mobile platforms without plugins.
- **File size**: AAC at 128kbps is roughly half the size of an equivalent MP3 and significantly better quality.

The `.m4a` extension is preferred for audio-only files. The `.mp4` extension is also acceptable and interchangeable. When users upload older files in other formats (MP3, WAV, OGG, FLAC), the backend should accept them as-is and store them in whatever format they arrive in — we do not need to force conversion of uploads, only of live recordings from the web client if the browser produces WebM.

## Features

### 1. Live Recording

- A **Record** button starts capturing audio from the device microphone.
- Visual feedback during recording: a timer showing elapsed time and a pulsing indicator.
- A **Stop** button ends the recording.
- After stopping, the user is prompted to fill in session metadata (see section 4) before or during upload.
- The audio file is uploaded to Amazon S3 in the background. A progress indicator is shown.
- Microphone permissions are requested at the point of first use.

**Platform notes:**
- iOS (native app): use `AVAudioRecorder` targeting `.m4a` output.
- Android (native app): use `MediaRecorder` with `MPEG_4` container and `AAC` encoder.
- Web: use the browser `MediaRecorder` API. Accept whatever format the browser produces and upload it directly. On Safari this will be M4A; on Chrome/Firefox this will be WebM. Both are stored as-is on S3.

### 2. Upload Past Sessions

- An **Upload** button opens the device file picker filtered to audio files.
- Accepted formats: M4A, MP4, MP3, WAV, FLAC, OGG, WEBM (any audio file the user has).
- After selecting a file, the user fills in session metadata before uploading.
- The upload goes directly to S3 (same multipart upload pattern used elsewhere in the app).
- Because users may be uploading recordings from years ago, the **date field defaults to today but can be freely overridden** (see section 4).

### 3. Session List

- Sessions are displayed as a list sorted by session date descending (newest first).
- Each list item shows: session name, date, duration, rating, and any tags.
- Tapping a session opens the session detail view.
- List can be filtered by tag or sorted by date, rating, or name.

### 4. Session Metadata

Each session stores the following:

| Field | Description |
|---|---|
| **Name / Title** | Short label for the session (e.g., "Tuesday Rehearsal", "Gig at The Rusty Nail"). Required. |
| **Session Date** | The date the recording was made. Defaults to today (for live recordings and same-day uploads) but can be overridden to any past date. Stored separately from `created_at` so the user can accurately date historical uploads. |
| **Duration** | Calculated automatically from the audio file. Read-only. |
| **Comments / Notes** | Free-text field for anything the user wants to note about the session — what songs were played, who was there, gear used, etc. No length limit. |
| **Tags** | A list of short labels. Examples: `rehearsal`, `gig`, `original`, `cover`, `electric`, `acoustic`, `full band`, `duo`. Tags are user-defined (not a fixed list). Multiple tags per session. |
| **Rating** | A numeric rating from **1–10**. This gives more granularity than a label system (Good/Bad/Best) while still being simple. A rough guide could be shown in the UI: 1–3 = needs work, 4–6 = decent, 7–9 = good, 10 = keeper. The rating is optional — sessions can be unrated. |

### 5. Session Detail / Playback

- Shows all metadata fields, all of which are editable after the fact.
- An in-app audio player with play/pause, seek bar, and elapsed/total time display.
- Shows the file format, file size, and upload date.
- A **Delete** button to remove the session and its S3 file.

## Storage

- Audio files are stored on Amazon S3 in a user-specific path, e.g.:
  `sessions/{userId}/{sessionId}/{filename}.m4a`
- Metadata (name, date, comments, tags, rating) is stored in the database.
- Playback uses a signed/proxied URL (same approach as other media in the app) rather than exposing the S3 URL directly.

### Proposed Database Table: `sessions`

| Column | Type | Notes |
|---|---|---|
| `id` | INT, PK, AUTO_INCREMENT | |
| `user_id` | INT, FK → users | Owner of the session |
| `title` | VARCHAR(255) | Required |
| `file_path` | VARCHAR(512) | S3 key |
| `file_format` | VARCHAR(20) | e.g. `m4a`, `mp3`, `webm` |
| `file_size` | BIGINT | Bytes |
| `duration_seconds` | INT | Calculated from file |
| `session_date` | DATE | User-set date of the recording |
| `notes` | TEXT | Free-text comments |
| `rating` | TINYINT | 1–10, nullable |
| `created_at` | TIMESTAMP | When the record was created in the DB |
| `updated_at` | TIMESTAMP | Last metadata edit |

### Proposed Database Table: `session_tags`

| Column | Type | Notes |
|---|---|---|
| `id` | INT, PK, AUTO_INCREMENT | |
| `session_id` | INT, FK → sessions | |
| `tag` | VARCHAR(100) | Lowercase, trimmed |

## API Endpoints (Proposed)

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/sessions` | List all sessions for the current user |
| `POST` | `/api/sessions` | Create a new session record (metadata only, before upload) |
| `POST` | `/api/sessions/upload` | Multipart upload — audio file + metadata in one request |
| `GET` | `/api/sessions/:id` | Get a single session with all metadata |
| `GET` | `/api/sessions/:id/stream` | Returns a signed S3 URL or proxied stream for playback |
| `PUT` | `/api/sessions/:id` | Update metadata (title, date, notes, tags, rating) |
| `DELETE` | `/api/sessions/:id` | Delete session record and S3 file |

## Future Features (Not in Scope for Initial Build)

The following are ideas for later phases:

- **Set list linking**: associate a session with a specific set list from the Lyric Lab tool.
- **Band / collaborator tagging**: tag other Breakroom users who participated in the session.
- **Waveform visualization**: display an audio waveform on the session detail page.
- **Trimming**: light in-app trimming to cut silence at the start/end of a recording.
- **Export / Share**: generate a shareable link or download the file.
- **Session groups / albums**: group multiple sessions under a named project or album.
- **Storage quota**: a per-user storage limit with a usage indicator.

## Open Questions

1. Should tags be drawn from a user's own history (autocomplete from past tags) or completely free-form with no suggestions? Autocomplete from past tags would be a nice UX improvement.
2. Should the rating scale (1–10) have named labels shown in the UI as hints, or just the number?
3. For the web app live recording: do we want to attempt in-browser format detection and normalize to M4A via a server-side ffmpeg step, or just store whatever the browser produces?
4. Should sessions be private by default (only visible to the owner) or is there a future social/sharing aspect to plan for now?
