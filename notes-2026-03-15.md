# Content Moderation Plan
**Date:** 2026-03-15
**Scope:** Web (Breakroom) first, Android to follow
**Driver:** Apple App Store compliance requirements

---

## Requirements Summary

1. A method for **filtering** objectionable content
2. A mechanism for users to **flag** objectionable content
3. A mechanism for users to **block** abusive users
   - Blocking notifies the developer
   - Blocking removes offending content from the blocker's feed instantly
4. Developer must act on reports **within 24 hours** (remove content + eject user)
5. Admin can **manually review** flagged content and **restore** it if not actually objectionable
6. All removals are **soft-deletes** (content hidden, not destroyed)
7. All ejections are **non-destructive** (user restricted, not deleted — reversible)

---

## Proposed Architecture

### 1. Database Changes

#### New Tables

**`content_flags`**
Tracks every user-submitted flag on a piece of content.
```
id, content_type (enum: post|comment|chat_message|profile), content_id,
flagged_by_user_id, reason (text, optional), status (enum: pending|approved|restored),
reviewed_by_user_id (null until admin acts), reviewed_at, created_at
```
- `pending` = flagged, content soft-deleted, awaiting admin review
- `approved` = admin confirmed it is objectionable, content stays hidden
- `restored` = admin determined it was not objectionable, content reinstated

**`user_blocks`**
Tracks one user blocking another.
```
id, blocker_user_id, blocked_user_id, created_at
UNIQUE (blocker_user_id, blocked_user_id)
```
When a block is created the blocked user's content is hidden from the blocker's feed. This also triggers a flag (with reason "blocked by user") so admin is notified.

**`content_filter_keywords`**
Admin-managed list of words/phrases that trigger automatic filtering.
```
id, keyword, is_active, created_at
```
Content matching any active keyword is soft-deleted automatically on submission and a flag record is created (flagged_by_user_id = NULL, meaning "auto-filter").

#### Modifications to Existing Tables

Add `is_hidden BOOLEAN DEFAULT FALSE` and `hidden_at TIMESTAMP NULL` to:
- `blog_posts`
- `blog_comments` (or equivalent comments table)
- `chat_messages`
- `users` (for profile-level moderation — already has a Restricted group mechanism, but a dedicated `is_suspended` flag avoids group-query overhead)

> **Note on ejection:** The existing `Restricted` group will be used for ejected users (move user to Restricted group). This is already in the DB and already limits what restricted users can do. No schema change needed for ejection itself — just a new `user_groups` row. Reversal = remove from Restricted group.

### 2. Backend Changes

#### New Route File: `backend/routes/moderation.js`

**Flag endpoints (authenticated users):**
- `POST /api/moderation/flag` — flag a piece of content
  Body: `{ content_type, content_id, reason? }`
  - Immediately sets `is_hidden = true` on the target content row
  - Inserts a `content_flags` record with status `pending`
  - Sends notification to all admin users (via the existing notification system)
  - Returns 200

- `POST /api/moderation/block/:userId` — block a user
  - Inserts into `user_blocks`
  - Creates a `content_flags` record for the blocked user (type: `user`, reason: "blocked")
  - Sends admin notification
  - Returns 200 (frontend hides content client-side immediately; see Frontend section)

- `DELETE /api/moderation/block/:userId` — unblock a user

- `GET /api/moderation/blocks` — get current user's block list

**Admin-only endpoints (require admin permission):**
- `GET /api/moderation/flags?status=pending` — list flagged content for review
  Returns flag record + the actual content + flagging user + content author

- `PUT /api/moderation/flags/:id/approve` — confirm content is objectionable
  - Sets flag status to `approved`
  - Ensures `is_hidden = true` on content
  - Optionally ejects the content author (see below)

- `PUT /api/moderation/flags/:id/restore` — content is fine, reinstate it
  - Sets flag status to `restored`
  - Sets `is_hidden = false` on content

- `POST /api/moderation/eject/:userId` — restrict a user
  - Adds user to Restricted group
  - Soft-hides all their content (`is_hidden = true`) across all content tables

- `POST /api/moderation/reinstate/:userId` — reverse an ejection
  - Removes user from Restricted group
  - Does NOT automatically restore their content (admin restores individual items)

**Admin keyword filter endpoints:**
- `GET /api/moderation/keywords`
- `POST /api/moderation/keywords`
- `DELETE /api/moderation/keywords/:id`

#### Middleware: `backend/middleware/contentFilter.js`

A reusable middleware function called on any route that creates content (post, comment, chat message). It:
1. Loads active keywords from DB (cached with a short TTL to avoid per-request queries)
2. Checks submitted text fields against the keyword list
3. If a match is found: allows the insert to proceed but immediately sets `is_hidden = true` and creates a `content_flags` record (auto-flagged)

#### Notification hook

When any flag or block is created, use the existing notification system to send a popup notification to all users in the Administrator group. The notification name will be "Content Report" and will link to the new admin moderation panel.

### 3. Frontend Changes (Web)

#### New Admin Section: Content Moderation (`/admin/moderation`)

A new page in the existing admin panel with two tabs:

**Flagged Content tab**
- Table of pending flags: content type, snippet of content, author, flagging user, date flagged, time since flagged (red if > 20 hours to prompt 24-hour compliance)
- Actions per row: **Approve Removal** | **Restore Content** | **Eject User**
- Resolved flags (approved/restored) visible in a separate sub-tab for audit trail

**Keyword Filter tab**
- List of active filter keywords with add/delete controls

#### User-facing: Flag Button

A small "Flag" option added to the action menu (or `...` overflow) on:
- Blog posts
- Blog comments
- Chat messages

Clicking opens a small dialog: optional reason field + confirm button. On submit, content disappears from view immediately (optimistic hide).

#### User-facing: Block Button

A "Block User" option on user profile pages and in the `...` overflow on content items.
When blocked:
- All content by that user vanishes from the current user's feed immediately (client-side filter using the block list)
- A flag is sent to the backend for admin notification

#### Content Feed Filtering

All feed-rendering components (blog list, chat, comments) will:
1. Filter out items where `is_hidden = true` (already handled server-side — the API simply won't return hidden content to non-admin users)
2. Filter out items authored by users in the current user's block list (returned with the `/api/moderation/blocks` response, cached in a Pinia store)

Admin users will see hidden content in moderation panel only — not inline in regular feeds.

### 4. Implementation Order

The following sequence minimises broken states at each step:

1. **Database migrations**
   Add `is_hidden` columns + `content_flags` + `user_blocks` + `content_filter_keywords` tables

2. **Backend moderation routes** (flags, blocks, keywords, eject/reinstate)

3. **Content filter middleware** — hook into existing post/comment/chat create routes

4. **Admin moderation panel** (frontend) — so admin can act on reports before user-facing UI ships

5. **User flag button** on blog posts, comments, chat messages

6. **User block button** on profiles and content

7. **Block list store + feed filtering** on frontend

8. **Keyword management UI** in admin panel

9. **Android** — mirror the flag and block UI in the mobile app

### 5. Open Questions / Decisions Needed

- **Email vs in-app notification for admin alerts?**
  Current plan uses the existing popup notification system. Should it also send an email to a developer address?

  Answer: it should also send an email to: ContentNotification@cherryblossomdevelopment.com.  The email should contain the flagged text or image

- **What content types does "block" apply to?**
  Plan covers blog posts, comments, chat. Does it also cover profile bios, artwork in the gallery, lyrics?

  Answer: it should cover everything mentioned above, bios, artwork, lyrics, blog posts, comments, chat, profile bios.  the content_flags table should even be able to accept content that we never thought about.  in the enum for flag type should be "other" and this should require a reason (when other is selected only) also the content_id would naturally be missing in the other case so null needs to be an accepted value

- **Auto-eject on flag count threshold?**
  e.g. if a user accumulates 3 approved flags, auto-eject them? Or always manual?

  Answer: auto eject them, later on we can un-eject them if we feel that is the correc thing to do (there needs to be an interface to support this)

- **Visibility of flag/block to the flagged/blocked user?**
  They should probably not be notified that they were flagged (to avoid retaliation), but they should see a generic "your content has been removed" message if their content is hidden.

  Answer: correct assumption

  Final note: We also need to keep track of users who flag content to ensure that this feature is not being abused.  There should be an area in the admin section for "Frequent Content Flaggers" which lists users who have flagged content by how often they have flagged it.  We need to have the ability to disable such users especially if they are abusing this feature.
